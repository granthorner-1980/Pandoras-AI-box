📋Instruction · Example

What a good CLAUDE.md or system prompt looks like in practice.

Instruction

What it is

Persistent context that applies silently to every AI session — a project's CLAUDE.md, or the system prompt sent on every Anthropic API call. You don't trigger it; it's just always there, shaping behaviour in the background.

Where it lives

Claude Code

  • Project-wide: CLAUDE.md at the repo root
  • Global (all projects): ~/.claude/CLAUDE.md
  • API: passed as system on each request

GitHub Copilot

  • Project-wide: .github/copilot-instructions.md at the repo root
  • Path-specific: *.instructions.md files with YAML frontmatter (e.g. apply only to src/api/**)
  • VS Code settings: github.copilot.chat.codeGeneration.instructions (and similar) in settings.json

Example

A condensed, well-structured project CLAUDE.md:

# Acme Web App

A Next.js storefront for Acme. Customer-facing checkout, search, and account pages.

## Stack

- Next.js 14 (App Router) with TypeScript strict mode
- Tailwind CSS for styling
- Prisma + Postgres
- Tested with Vitest (unit) and Playwright (e2e)

## Conventions

- Components live in `src/components/` and use named exports only
- API routes live in `src/app/api/` — always validate input with Zod schemas in `src/schemas/`
- No `any` types — if you need a placeholder, use `unknown` and narrow

## Things to never do

- Don't introduce new state libraries (we use Zustand only)
- Don't use `useEffect` for data fetching — use Server Components or React Query
- Don't bypass auth middleware — every API route must call `requireUser()`

## Testing

Before declaring a task done, run:

```
pnpm typecheck
pnpm test
```

If the change touches checkout, also run `pnpm e2e:checkout`.

You can also see this repo's own CLAUDE.md for a real working example.

What makes this good
  • Top line tells the AI what the project is in one sentence — orienting it instantly.
  • Stack section is short and concrete — versions, languages, key libraries.
  • Conventions are specific ("named exports only", not "follow good practices").
  • "Things to never do" prevents common drift back to defaults the AI would otherwise pick.
  • Testing block gives a reproducible done-check, not vague "make sure it works."
  • No filler. No "be helpful." No restating what the code already shows.
Watch out for
  • Bloat costs tokens. Every line is sent on every message. Keep it focused — if you've never had to remind the AI of something, don't write it down.
  • Don't put fast-changing context in. Today's task state, current branch, current sprint goal — those belong in a Prompt, not here.
  • Don't restate what the code shows. If tsconfig.json already says strict mode is on, you don't need a line saying so.
  • Write it for the AI, not for humans. No "welcome to the project" tone. Direct, imperative, terse.
  • Update it when conventions change. A stale CLAUDE.md silently pushes the AI toward outdated patterns.
