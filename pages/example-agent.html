Agent

What it is

A focused sub-agent with its own system prompt, model, and tool access. The main Claude delegates work to it when the task description matches. You give it a goal — it figures out the steps, runs the tools, and returns a result.

Where it lives

Claude Code

  • Project: .claude/agents/<name>.md
  • Global: ~/.claude/agents/<name>.md

GitHub Copilot (Custom Agents)

  • Project: .github/agents/<name>.agent.md
  • User profile: ~/.copilot/agents/<name>.agent.md
  • Note: previously called "chat modes" — old .chatmode.md files should be renamed to .agent.md

Example

A code-explorer agent — fast, cheap, read-only codebase navigator:

---
name: code-explorer
description: Quickly locate and explain a feature in the codebase. Use for "where is X defined?" or "how does Y work?" questions. Read-only — does not modify files.
model: haiku
tools: Read, Grep, Glob, Bash
---

You are a read-only code exploration agent. Given a feature, function,
or symbol name, find it in the codebase and explain how it works.

Process:
1. Use Grep to find references — start broad, narrow down
2. Use Read to inspect the key files
3. Trace the flow end-to-end (caller → callee, data in → data out)
4. Return a concise explanation with `file:line` citations

Constraints:
- Do not modify any files (no Edit, no Write — your toolset reflects this)
- Do not run destructive shell commands
- Keep the response under 300 words
- If the feature spans many files, show the entry point and the
  2-3 most important hops, not exhaustive coverage

Output format:
**Summary:** one-line description
**Entry point:** file:line
**Key files:**
- file:line — what it does
- file:line — what it does
**Notes:** any gotchas worth knowing
What makes this good
  • Specific description. The main agent reads this to decide when to delegate. "Use for 'where is X' questions" is a clear trigger; "explores code" is not.
  • Cheap model. model: haiku — read-only navigation doesn't need Opus. Cost-conscious by design.
  • Restricted toolset. No Edit or Write means it physically can't modify files, even if the prompt slipped.
  • Concrete process. A 4-step recipe gives the agent a clear path to follow.
  • Hard constraints. Word limit, no destructive commands, no exhaustive coverage — keeps it focused.
  • Structured output. Predictable shape so the main agent can consume the result reliably.
Watch out for
  • Don't grant more tools than needed. Fewer tools = safer + cheaper + faster. An exploration agent doesn't need Edit.
  • The description controls delegation. If it's vague, the main agent won't know when to use this — and you'll never see it triggered.
  • Agents amplify mistakes. A vague goal becomes a confidently wrong multi-step workflow. Be specific about success criteria.
  • Long-running agents can loop or hallucinate paths. Set output limits and step limits.
  • Read-only by default. If you can achieve the task without write access, omit Edit/Write tools entirely.
  • Don't reach for an agent when a prompt would do. Agents are slower and harder to predict — only escalate when the task genuinely needs autonomy.
